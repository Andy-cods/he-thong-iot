#### 1.2.4 Order code generator — chọn gì?
**Format:** `SO-YYMM-####` (4-digit sequence reset hàng tháng).

3 cách sinh:
1. **Postgres sequence** reset monthly qua cron → **chọn** (simple, atomic, không race).
2. `pg_advisory_lock(hash(yearMonth))` + COUNT(*) + INSERT → phức tạp hơn, không gain gì.
3. UUIDv7 → user không đọc được, bỏ.

**Quyết định:** Dùng **1 sequence global `sales_order_no_seq`** + format `SO-${yymm}-${lpad(nextval,4)}`. Chấp nhận số sequence có thể nhảy tháng (SO-2604-0001 rồi SO-2605-0002) — trade-off để tránh cron reset phức tạp. Nếu khách chê, V1.3 bẻ sang composite key (monthly reset).

#### 1.4.2 Recursive CTE explode (rút từ research report phần 1 §Y + điều chỉnh cho schema thực)
```sql
INSERT INTO app.bom_snapshot_line (
  id, snapshot_id, order_id, source_bom_line_id, parent_snapshot_line_id,
  parent_item_id, child_item_id, level, depth, line_path, find_no, position_no,
  source_type, phantom_flag, substitute_group_code,
  qty_per, scrap_pct, required_qty, gross_required_qty,
  line_status, frozen_at
)
WITH RECURSIVE exploded AS (
  -- Anchor: level-1 lines của revision (parent_revision_line_id IS NULL)
  SELECT
    gen_random_uuid()                                            AS new_id,
    $1::uuid                                                      AS snapshot_id,
    $2::uuid                                                      AS order_id,
    brl.id                                                        AS source_bom_line_id,
    NULL::uuid                                                    AS parent_snapshot_line_id,
    NULL::uuid                                                    AS parent_item_id,
    brl.component_item_id                                         AS child_item_id,
    1                                                             AS level,
    1                                                             AS depth,
    LPAD(brl.position::text, 3, '0')                              AS line_path,
    brl.position                                                  AS find_no,
    brl.position                                                  AS position_no,
    COALESCE(i.default_source_type, 'BUY')                        AS source_type,
    brl.phantom_flag,
    brl.substitute_group_code,
    brl.qty_per_parent                                            AS qty_per,
    brl.scrap_percent                                             AS scrap_pct,
    (brl.qty_per_parent * $3::numeric)                            AS required_qty,
    (brl.qty_per_parent * $3::numeric * (1 + brl.scrap_percent))  AS gross_required_qty
  FROM app.bom_revision_line brl
  JOIN app.item i ON i.id = brl.component_item_id
  WHERE brl.revision_id = $4::uuid AND brl.parent_revision_line_id IS NULL

  UNION ALL

  SELECT
    gen_random_uuid(), e.snapshot_id, e.order_id,
    child.id, e.new_id, e.child_item_id, child.component_item_id,
    e.level + 1, e.depth + 1,
    e.line_path || '/' || LPAD(child.position::text, 3, '0'),
    child.position, child.position,
    COALESCE(i.default_source_type, 'BUY'),
    child.phantom_flag, child.substitute_group_code,
    child.qty_per_parent, child.scrap_percent,
    (e.required_qty * child.qty_per_parent),
    (e.gross_required_qty * child.qty_per_parent * (1 + child.scrap_percent))
  FROM exploded e
  JOIN app.bom_revision_line child ON child.parent_revision_line_id = e.source_bom_line_id
  JOIN app.item i ON i.id = child.component_item_id
  WHERE e.depth < 5   -- BOM_MAX_LEVEL = 5 (same DB constraint bom_line)
)
SELECT
  new_id, snapshot_id, order_id, source_bom_line_id, parent_snapshot_line_id,
  parent_item_id, child_item_id, level, depth, line_path, find_no, position_no,
  source_type, phantom_flag, substitute_group_code,
  qty_per, scrap_pct, required_qty, gross_required_qty,
  'PLANNED', now()
FROM exploded
WHERE phantom_flag = false   -- phantom không insert line, nhưng children của nó đã duyệt qua
ORDER BY line_path;
```

**Params:** `$1=snapshotId, $2=orderId, $3=rootQty, $4=revisionId`.

#### 1.4.3 Thực tế triển khai — 4 điểm phải cẩn thận
1. **Default source_type:** schema `app.item` V1 **chưa có** `default_source_type`. → **Thêm** cột này trong migration 0005 (enum `BUY`/`MAKE`, default `BUY`). Fallback nếu null = `BUY`.
2. **Scrap cumulative chính xác:** CTE dùng multiplication `(1 + scrap)` mỗi cấp → OK cho xưởng cơ khí. Không dùng sum additive sai (nhiều nguồn tài liệu nhầm chỗ này).
3. **Phantom skip insert nhưng children vẫn chạy:** CTE duyệt qua phantom (UNION ALL join vào child) sau đó filter `WHERE phantom_flag = false` tại SELECT cuối. `parent_snapshot_line_id` cho children-of-phantom = ID snapshot line của ancestor gần nhất **không phải phantom** (V1.2 simplification: assume phantom 1 cấp, V1.3 mới đúng triệt để).
4. **Deterministic ordering:** `line_path` dùng `LPAD(position, 3, '0')` → chuỗi "001/002/003" sort chính xác đến 999 sub-component mỗi cấp. Vượt 999 → nổ (đặt CHECK trên bom_line `position <= 999`).

#### 1.5.6 Concurrent race — optimistic locking
- Dùng `version_lock` column tăng mỗi UPDATE.

### 1.7 Module 6 — Shortage Board aggregate (2 ngày)

#### 1.7.1 Query core
```sql
-- /api/shortage/items
SELECT
  bsl.child_item_id AS item_id,
  i.sku, i.name,
  SUM(bsl.gross_required_qty) AS total_required,
  SUM(bsl.qc_pass_qty)         AS total_available_alloc,
  SUM(bsl.open_purchase_qty)   AS total_on_order,
  SUM(bsl.remaining_short_qty) AS total_short,
  COUNT(DISTINCT bsl.snapshot_id) AS order_count,
  MAX(so.due_date)             AS latest_due_date
FROM app.bom_snapshot_line bsl
JOIN app.bom_snapshot bs ON bs.id = bsl.snapshot_id
JOIN app.sales_order so ON so.id = bs.order_id
JOIN app.item i ON i.id = bsl.child_item_id
WHERE bsl.line_status IN ('PLANNED','PURCHASING','INBOUND_QC')
  AND so.status NOT IN ('CANCELLED','CLOSED')
  AND bsl.remaining_short_qty > 0
GROUP BY bsl.child_item_id, i.sku, i.name
HAVING SUM(bsl.remaining_short_qty) > 0
ORDER BY SUM(bsl.remaining_short_qty) DESC;
```

**Performance:** Index partial `WHERE remaining_short_qty > 0` đã có từ Module 3. Test với 1000 snapshot × 100 line trung bình = 100k row → expect < 300ms.

**Nếu chậm > 500ms:** Materialized view `mv_shortage_by_item` refresh every 5 minute bằng cron job BullMQ (không refresh on-demand tránh cache stampede).

#### 1.7.2 API endpoints (2)
- `GET /api/shortage/items` — aggregate by item (query trên)
- `GET /api/shortage/by-order/[orderId]` — breakdown cho 1 order

### 1.8 Module 7 — Receiving real inventory update (2 ngày)

#### 1.8.1 Flow hiện tại (V1.1-alpha stub)
User scan `/pwa/receive/[poId]` → Dexie queue → POST `/api/receiving/events` → chỉ insert `receiving_event` + audit, **chưa** tạo `inventory_txn`, **chưa** update inventory.

#### 1.8.2 Flow mới V1.2
```
Scan → receiving_event insert
     → inbound_receipt (hoặc existing) + inbound_receipt_line insert
     → inventory_lot_serial (tìm hoặc tạo mới theo lot_code + item_id)
     → inventory_txn insert (tx_type=IN_RECEIPT, qty, lot_serial_id, ref=receiving_event.id)
     → UPDATE purchase_order_line SET received_qty += qty
     → IF all PO line đã đủ → PO status PARTIAL → RECEIVED
     → UPDATE bom_snapshot_line SET received_qty += qty (qua po_line.linked_snapshot_line_id)
     → IF QC auto-pass policy (V1.2 default PENDING): không transition state tự động
       ELSE: transition PURCHASING → INBOUND_QC
```

#### 1.8.3 Schema delta
- Thêm `receivingEvent.inventoryTxnId` UUID reference (nullable, fill sau post).
- Thêm `receivingEvent.inboundReceiptLineId` UUID reference.
- Không cần bảng mới — chỉ link 3 bảng có sẵn.

#### 1.8.4 QC flow minimal
- `inbound_receipt.qc_flag` có sẵn enum `PENDING | PASS | FAIL`.
- V1.2: sau scan → `PENDING`. QC user vào `/receiving/qc/[receiptId]` đánh dấu PASS/FAIL.
- PASS → transition snapshot line INBOUND_QC → AVAILABLE, update `qc_pass_qty`.
- FAIL → hold lot (thêm cột `inventory_lot_serial.status` varchar HOLD/OK, default OK), không cho reserve. Transition snapshot line INBOUND_QC → PLANNED (loop về, đợi PO replacement).

#### 1.8.5 API endpoints (3)
- `POST /api/receiving/events` — **enhance** hiện tại để post đầy đủ chain (transaction 1 lần, atomic).
- `POST /api/receiving/qc/[receiptId]` — body `{decision: PASS|FAIL, notes}`. Side effect: transition toàn bộ line của receipt.
- `GET /api/inventory/on-hand` — aggregate on-hand qty per item (SUM txn qty theo tx_type dấu).

#### 1.8.6 UI
- Enhance `/pwa/receive/[poId]` — thêm banner "QC pending X lot" + link.
- Route mới `/receiving/qc` — list receipt QC_PENDING.
- Route mới `/receiving/qc/[receiptId]` — detail + 2 button PASS/FAIL.

#### 1.8.7 Over-delivery
PO line 100, nhận 110 → allow with warning toast. `received_qty` = 110. Không block để không kẹt flow vật lý (hàng đã đến kho). Audit log capture.

## §6 Performance concerns — 6 điểm cứng

1. **Recursive CTE 1000 node < 5s.** Benchmark bắt buộc trên VPS (2 vCPU/2GB) không phải local (8 core). Chuẩn bị case 2000+10 depth để break threshold.
2. **Generated column `remaining_short_qty` STORED.** Mỗi UPDATE 1 trong 5 qty col trigger recompute + index update. Test concurrent UPDATE 100 line/s không bị deadlock (test script BullMQ fan-out).
3. **Snapshot Board virtualize.** `@tanstack/react-virtual` đã dùng. Virtualize threshold 50 row. Dưới 50 render full (ux mượt hơn).
4. **State transition concurrent race.** Optimistic lock `version_lock`. Test 10 user cùng transition 1 line: 1 pass, 9 get 409.
5. **Shortage aggregate query slow.** Threshold 500ms → materialized view refresh 5m. Cron dùng BullMQ `shortage-refresh` repeatable job.
6. **Dashboard cache invalidate.** Khi snapshot line qty change (qua receiving / QC / reservation) → invalidate `dashboard:overview` key Redis. Hook đã có từ V1.1-beta mutation item/bom, thêm hook mới cho snapshot mutation.

## §9 Decision checklist — 20 decision cho planner chốt

| # | Decision | Đề xuất | Lý do |
|---|---|---|---|
| D1 | Order code format | `SO-YYMM-####` dùng Postgres sequence global (không reset monthly) | Simple atomic, trade-off số sequence nhảy tháng |
| D2 | Snapshot immutable sau explode? | **YES**, chỉ state + qty col update được, không sửa structure | Bằng chứng lịch sử + consistency |
| D3 | 10-state transition ai làm được? | planner + admin. `operator` chỉ PURCHASING → INBOUND_QC (scan receiving) | RBAC minimum V1.2 |
| D4 | Recursive CTE depth limit? | **5** (same BOM_MAX_LEVEL của bom_line DB CHECK) | Consistent, defer 10+ V1.4 |
| D5 | PR auto-suggest từ shortage? | **YES** batch — user chọn từ shortage table → "Tạo PR" prefill, không auto-confirm | User control cuối |
| D6 | PO multi-supplier? | **NO** V1.2, 1 PO = 1 supplier. PR nhiều item tách N PO theo `preferred_supplier_id` | Simplify |
| D7 | Receiving qty > expected? | **Allow với warning**, không block | Không kẹt flow vật lý |
| D8 | QC NG lot → auto-block? | **YES** auto (cột `inventory_lot_serial.status = HOLD`), không cho reserve | Tránh pick nhầm |
| D9 | Shortage Board refresh? | Client poll 60s, SSE defer V1.3 | YAGNI |
| D10 | Snapshot sync vs async threshold | `total_lines_estimate > 500` → async BullMQ | Benchmark cụ thể |
| D11 | `remaining_short_qty` generated STORED hay trigger? | **Generated STORED** (Postgres 16 support) | Đơn giản hơn trigger |
| D12 | `scrap_pct` cumulative công thức? | `(1+scrap₁) × (1+scrap₂) × ...` multiplicative | Chính xác hơn sum |
| D13 | Phantom multi-level V1.2? | **1 cấp** only V1.2, defer V1.3 nested | Simplify CTE |
| D14 | Substitute V1.2? | Explode **primary only** (item đầu theo position trong substitute_group) | UI picker defer V1.3 |
| D15 | Admin override state transition? | **Có**, trừ `CLOSED → *` (final state), log audit `admin_override=true` | Recovery path |
| D16 | State RESERVED → AVAILABLE cần guard nào? | Xác nhận reason + không có WO active link đến line này | Avoid accidental release |
| D17 | BOM Revision release trùng tree? | Reject 409 `tree_hash` duplicate | Không spam rev |
