## §1 BOM Tree Editor — Tech Stack + UX

### D1. Tree library: `@dnd-kit/core` hay `react-arborist` hay tự cook?
**CHỐT:** `@dnd-kit/sortable` + `@dnd-kit/utilities` + tự quản state tree trong Zustand. KHÔNG dùng `react-arborist`.
**WHY:** `react-arborist` lock style + virtualization opinionated, khó custom inline-edit 8 field/row; dnd-kit đã có trong V1.0 Items DataTable reorder → reuse giảm bundle.

### D2. Drag-drop validation: cho phép move component qua parent khác?
**CHỐT:** Cho phép move giữa bất kỳ 2 parent CÙNG revision DRAFT. BLOCK move nếu: (a) target là descendant của source (cycle), (b) target là phantom đã có child khác source_type.
**WHY:** Move cross-parent là use case thật (refactor sub-assembly); cycle check DFS visited set trước khi commit op, O(depth) đủ cho BOM 500 node.

### D3. Seq management: auto-rebalance khi insert giữa?
**CHỐT:** `find_no` dùng integer × 10 gap (10, 20, 30...). Insert giữa → dùng midpoint (15, 25). Khi hết gap (chênh lệch < 2) → auto-resequence toàn bộ siblings về 10/20/30.
**WHY:** Integer thay float 0.5 vì DB `find_no INT` (schema), đơn giản, resequence 1 batch update + audit log dễ. 10-gap cho phép 9 insert liên tiếp trước khi cần re-balance.

### D4. Undo/redo: có cần không? Dùng gì?
**CHỐT:** Có, tự cook `undoStack: Operation[]` trong Zustand store, limit 50 ops, KHÔNG dùng library như `zundo`.
**WHY:** Spec editor đã nói Ctrl+Z/Y + 50 ops; 3 op type (add/edit/move/delete) quá đơn giản để import lib 3kb. Persist vào sessionStorage để survive accidental reload.

### D5. Save strategy: auto-save per-edit hay explicit button?
**CHỐT:** Dual: (a) autosave debounce 2s per-field edit (PATCH 1 component), (b) explicit "Lưu draft" button batch toàn bộ dirty ops (tối đa 50).
**WHY:** Autosave giảm mất data; explicit button cho power-user muốn atomic commit. Backend API đã có per-component PATCH + batch reorder — reuse.

### D6. Concurrent edit 2 user: LWW vs optimistic lock?
**CHỐT:** Optimistic lock với `updated_at` per-component. PATCH gửi `if_updated_at=<timestamp>`, server 409 nếu mismatch + return server state. UI show dialog diff "User khác đã sửa, [Reload] [Giữ bản của tôi]".
**WHY:** Last-write-wins silent mất work design engineer; 409 + dialog rõ ràng. Không cần row-level lock vì conflict hiếm (1-2 design_engineer max).

### D7. Large BOM (500+ nodes): virtualize? Lazy load?
**CHỐT:** V1.1: render full tree flat (no virtualize) với collapse default tất cả node depth ≥ 2. 500 node DOM = ~2MB memory, acceptable. Virtualize + lazy-load defer V1.2 khi có BOM > 1000 node thực.
**WHY:** Song Châu BOM thực < 200 node (Risk R1.2), YAGNI. Virtualize + collapse kết hợp phức tạp (keep-alive expanded state on scroll), tốn 2-3 ngày không justify.

## §2 BOM Revision — Release Flow + Immutability

### D8. "Release R01" transaction: 1 TX hay 2 TX?
**CHỐT:** 1 TX duy nhất. `BEGIN; UPDATE bom_revision SET status='RELEASED'; INSERT state_transition_log; UPDATE siblings SET status='OBSOLETE' WHERE template_id=X AND status='RELEASED' AND id!=me; COMMIT;`. KHÔNG copy rows sang snapshot riêng — snapshot chỉ tạo khi release ORDER (xem §3).
**WHY:** Revision release KHÔNG copy data (bom_component rows đã immutable qua trigger `ck_bom_rev_released`). Snapshot = immutable copy per-ORDER, không per-revision. Tách biệt discipline.

### D9. Release rollback nếu lỗi giữa chừng?
**CHỐT:** Postgres TX rollback tự động rollback tất cả. Service layer catch exception → rollback + log `audit.state_transition_log` với status='RELEASE_FAILED' + error message. UI show toast danger + stay DRAFT.
**WHY:** 1 TX = atomic, không cần cleanup manual. Log failed attempt để debug (design engineer retry).

### D10. Revision number generator: per-template hay global?
**CHỐT:** Per-template sequential `R01, R02, R03...` với UNIQUE constraint `(template_id, revision_no)`. Generator: `SELECT COALESCE(MAX(CAST(SUBSTRING(revision_no FROM 2) AS INT)),0)+1 FROM bom_revision WHERE template_id=X FOR UPDATE` → format `R%02d`.
**WHY:** `FOR UPDATE` lock row template → concurrency safe với 2 design_engineer clone cùng lúc. Global sequential (R0001) confusing UX, per-template match chuẩn PLM (Odoo, Epicor).

### D11. Release permission: ai được release?
**CHỐT:** `design_engineer` + `admin` + `manager`. RBAC R2 §3 matrix: design_engineer có `E` trên BOM/Revision = tạo+sửa+release. NOT `material_planner` (chỉ V).
**WHY:** Release revision = engineering decision, planner không có authority. Manager có `V` nhưng V1.1 cho manager release để unblock khi DE vắng (acceptable override).

### D12. Unrelease (RELEASED → DRAFT) có cho phép không?
**CHỐT:** KHÔNG. Once RELEASED = immutable forever. Nếu muốn sửa → tạo revision mới R02 từ R01. Button "Unrelease" KHÔNG xuất hiện trong UI V1.1.
**WHY:** Discipline bất biến (R2:245). Unrelease phá reference integrity của snapshot đã tạo (source_bom_revision_id). DB trigger enforce UPDATE revision_status RELEASED→DRAFT RAISE EXCEPTION.

### D13. Compare 2 revision diff (R01 vs R02): V1.1 cook hay defer?
**CHỐT:** **DEFER V1.3** (đã liệt kê D9 §13 plan). V1.1 chỉ hiển thị số lines change: "R02 có 27 component, R01 có 25 component (+2 -0 mod 3)".
**WHY:** Diff UI cần tree-diff algorithm (LCS variant) + side-by-side highlight — 3 ngày dev. Không blocking V1.1 acceptance. Stub summary đủ cho PO review.

## §3 Order Entry + Snapshot Creation

### D14. Order code generator: per-year hay global?
**CHỐT:** Format `SO-YYMM-####` per-month sequential (vd `SO-2604-0001`). Generator: Postgres sequence `sales.order_no_seq_202604` created on-demand. Hoặc đơn giản hơn: `SELECT MAX(...) + 1 FOR UPDATE` như D10.
**WHY:** Per-month reset dễ đọc cho sales team (biết đơn tháng nào). Global sequential quá abstract. Fallback MAX+1 trong TX đủ concurrency an toàn cho < 100 order/day.

### D15. Snapshot đồng bộ hay async? Threshold?
**CHỐT:** Hybrid. Nếu dry-run count(bom_component chain) < 100 node → sync POST response 200 kèm snapshot data (< 2s). Nếu ≥ 100 node → async enqueue BullMQ, response 202 + `jobId`, UI poll 2s.
**WHY:** 25 component × 3 qty = 75 line < 2s, UX tốt hơn polling. > 100 node có thể chạm 5s threshold → async tránh HTTP timeout. Threshold config env `SNAPSHOT_SYNC_MAX_NODES=100`.

### D16. Async: BullMQ config?
**CHỐT:** Queue `bom-explode`, `jobId = snapshot-<snapshotId>` (deterministic idempotent), retry 3 × exponential backoff 1s/5s/30s, removeOnComplete { count: 100, age: 24h }, removeOnFail { count: 500 }.
**WHY:** jobId deterministic → client retry POST `/release-snapshot` không tạo duplicate job. TTL 24h complete để debug, 500 failed retain cho post-mortem.

### D17. Snapshot timeout: kill + rollback?
**CHỐT:** Worker timeout 60s (hard). Nếu timeout: (a) worker rollback TX insert snapshot_line, (b) UPDATE `bom_snapshot.status='FAILED'`, (c) UPDATE `customer_order.released_snapshot_at=NULL`, (d) log error context, (e) UI toast "Explode timeout, thử lại hoặc liên hệ IT".
**WHY:** 60s = 12× threshold 5s, gần như chắc chắn BOM có cycle hoặc > 5000 node pathological. Rollback giữ DB clean. Order quay về CONFIRMED để user retry.

### D18. Order có > 1 snapshot (re-snapshot sau ECO)?
**CHỐT:** V1.1: **1 snapshot per order_line** (1 order có N line = N snapshot). KHÔNG cho re-snapshot. Order đã `released_snapshot_at IS NOT NULL` → button "Release snapshot" disabled. Re-snapshot defer V1.2 (cần versioning snapshot).
**WHY:** Plan §13 D15 đã defer snapshot rollback V1.2. V1.1 giữ discipline bất biến tuyệt đối. Nếu sai snapshot → cancel order + tạo order mới (UX painful nhưng đúng V1.1).

### D19. Partial snapshot: user chọn 50% sub-assembly?
**CHỐT:** **KHÔNG** — YAGNI. V1.1 release snapshot = explode ALL lines của ALL order_lines. Không có partial picker.
**WHY:** Partial snapshot cần UI tree picker + delta logic khi thêm sau → phức tạp không justify. Nếu user muốn ship 50% trước → dùng `partial_delivery_allowed=true` + V1.3 shipment split.

## §4 Recursive CTE BOM Explode

### D20. Phantom expansion: merge hay giữ flag riêng?
**CHỐT:** Recursive CTE **duyệt qua phantom** để đi xuống children nhưng **KHÔNG INSERT row** cho phantom (clause `WHERE phantom_flag=false` ở SELECT cuối). Children của phantom có `parent_snapshot_line_id` = ancestor gần nhất non-phantom. qty rollup = qty_per_parent × phantom.qty_per_parent × ... (chain multiplication tự động).
**WHY:** Phantom = "không lưu kho" theo spec, nhưng qty phải propagate. Giữ riêng với flag `is_phantom_pass_through` redundant vì `parent_snapshot_line_id` đã đủ truy ngược. Plan §6.3 đã chốt, confirm.

### D21. Substitute: V1.1 chỉ primary — rule pick?
**CHỐT:** Primary = component đầu tiên theo `find_no` ASC trong cùng `(parent_component_id, substitute_group_code)`. Store `substitute_group_code` vào snapshot_line để V1.2 reservation chọn alternate. Alternate components KHÔNG tạo snapshot_line (skip) ở V1.1.
**WHY:** Plan §6.4 confirm. "Primary" cần determiniistic → find_no ASC clear. Không cần thêm column `is_primary` (implicit min find_no).

### D22. Scrap %: per-level multiply hay rollup cuối?
**CHỐT:** Per-level multiply: `gross_required = required × ∏(1 + scrap_pct_ancestor_chain)`. Formula cumulative = simple sum (`Σ scrap_pct`) **KHÔNG** chính xác 100% nhưng acceptable cho scrap < 5% (error < 0.25%).
**WHY:** Plan §6.5 chốt simple sum. Probability compound `1-∏(1-p)` defer V1.2. Xưởng cơ khí scrap typical 1-3% → error âm bỏ qua được.

### D23. Circular reference: pre-validate DFS hay tin CTE?
**CHỐT:** Defense in depth. (a) UI validate khi drop node: DFS visited set check cycle trước khi POST. (b) Service `releaseRevision` chạy DFS full revision trước khi set RELEASED (block release nếu cycle). (c) CTE có `WHERE depth < 10` làm safety net cuối. (d) Unique constraint `(revision_id, parent_item_id, child_item_id)` giúp catch self-loop.
**WHY:** CTE infinite loop gây worker hang 60s rồi timeout — đắt. DFS O(V+E) < 10ms cho 500 node, preventive check đáng giá.

### D24. Negative qty: reject hay support?
**CHỐT:** **REJECT.** DB CHECK constraint `qty_per_parent > 0` (đã có trong plan §2.1.2). Zod schema `z.number().positive()`.
**WHY:** Scrap returnable = concept V2+ (reverse logistics). V1.1 xưởng cơ khí không có use case qty âm. Cho phép âm = bẫy tính toán gross_required sai.

### D25. Performance benchmark: VPS hay local?
**CHỐT:** Benchmark trên cả 2. (a) Local Windows dev: baseline, target < 2s cho 1000 node. (b) VPS 2 vCPU production: target < 5s. Nếu VPS fail → enable async worker fallback (đã có D15).
**WHY:** VPS 2 vCPU đã có nhiều I/O contention (postgres + caddy + worker) → worst-case test đúng môi trường real. Local nhanh hơn 2-3× là normal baseline.

## §5 State Machine — 10 State × 20 Transition

### D26. Transition trigger: manual hay event-driven?
**CHỐT:** V1.1: **cả 2**. (a) Manual qua `POST /api/snapshot-lines/[id]/transition` (admin override + test). (b) Auto-trigger placeholder: hàm `transitionSnapshotLine()` được export, V1.2/V1.3 sẽ gọi từ service `postGrn`, `postQcDecision`, `confirmPick`... Hiện tại V1.1 chỉ có manual gọi để test + dashboard stub.
**WHY:** V1.1 chưa có event source thực (PO/QC/Receipt chưa tồn tại). Manual + hàm export = đủ unblock downstream, sẵn sàng cho V1.2 wire-in.

### D27. Guard condition location: Zod/DB/service/all?
**CHỐT:** 3 layers:
- **Zod:** validate `toStatus` enum (type safety trước khi vào service).
- **Service** (`isValidTransition` trong `bom-snapshot-line.ts`): business rule transition matrix + role check (defense in depth).
- **DB CHECK:** `ck_snap_line_status` enum only (NOT transition rule — DB không biết fromStatus khi INSERT).
Trigger DB transition KHÔNG dùng — complicated maintenance.
**WHY:** Zod = fail-fast UX; service = business logic centralized; DB = data integrity. 3 layers phù hợp defense-in-depth OWASP.

### D28. Admin override: skip state cho phép không?
**CHỐT:** **Có**, admin được phép mọi transition TRỪ `CLOSED → *` (final state absolute). Logic plan §5.4 đã code đúng: `if (userRole === 'admin' && from !== 'CLOSED') return { valid: true }`.
**WHY:** Admin cần unblock khi dev sai logic hoặc seed data. CLOSED final vì audit + billing integrity (V1.3 costing snapshot).

### D29. State rollback ISSUED → RESERVED?
**CHỐT:** **Có** cho recall scenario. Transition `ISSUED → RESERVED` trong matrix. Side effect: `issued_qty -= recallQty, reserved_qty += recallQty`. Require reason text mandatory (Zod min 10 char). Audit log bắt buộc.
**WHY:** Warehouse recall ngược pick case thực (sai lô, sai bin). Audit reason mandatory = compliance. Plan §5.2 đã có `ISSUED → RESERVED` nhưng chưa note reason mandatory.

### D30. Concurrent state change 2 user: lock hay optimistic?
**CHỐT:** Pessimistic row lock trong TX: `SELECT ... FROM bom_snapshot_line WHERE id=X FOR UPDATE` trước khi UPDATE status. 2 user click Advance cùng lúc → user thứ 2 block ~ms, re-check state sau khi lock → 422 nếu fromStatus đã đổi.
**WHY:** Optimistic 409 UX xấu cho planner (click xong báo lỗi). Lock 50ms acceptable. Postgres row-lock release khi TX commit.

## §6 Partial Qty 9 Cột

### D31. `remaining_short_qty`: generated column hay trigger?
**CHỐT:** Postgres `GENERATED ALWAYS AS (...) STORED` (đã có plan §2.1.3). KHÔNG dùng trigger.
**WHY:** Generated STORED = auto update on INSERT/UPDATE cùng TX, index được, 0 code maintenance. Trigger thêm failure mode + perf overhead. Drizzle unsupport thì TS type readonly qua `.$type<never>()`.

### D32. Decimal precision `numeric(18,4)` đủ?
**CHỐT:** **numeric(18,6)** (đã plan §2.1.3). 6 decimals đủ cho gram (0.001 kg = 1g = 0.000001 tấn). Amount money dùng `numeric(18,2)` riêng.
**WHY:** Plan đã chốt `numeric(18,6)` cho qty. VND tiền tệ không cần > 2 decimal. 18 digit integer part = 999 tỷ qty, dư thừa.

### D33. Over-receive (received > required): hiển thị gì?
**CHỐT:** Cho phép received > gross_required (warehouse thực tế có over-delivery vendor). `remaining_short_qty = GREATEST(..., 0)` = 0 clamp (không âm). UI column "Đã nhận" hiển thị `received_qty` + badge cam "⚠ Over +5" nếu received > gross_required.
**WHY:** Reject over-receive = UX ngu, vendor giao thừa phải cho phép ghi. Badge cam warning cho purchasing liên hệ vendor. Invariant I2 vẫn thoả (received ≥ qc_pass).

### D34. Reservation chỉ từ `qc_pass_qty` hay cả `received_qty`?
**CHỐT:** **Chỉ `qc_pass_qty`.** Rule: `reserved_qty ≤ qc_pass_qty - (issued_qty + assembled_qty)` (Invariant I3). Received chưa QC = chưa sẵn sàng, không reserve được.
**WHY:** R2 §247 + Song Châu nghiệp vụ QC incoming mandatory. Reserve hàng chưa QC = risk khi QC FAIL phải release reservation lần nữa. Discipline chặt.

## §8 Dashboard Real (Thay Mock V1.0)

### D38. `/api/dashboard/overview`: 1 endpoint hay 4?
**CHỐT:** **1 endpoint** trả full payload (đã plan §3.7). Cache Redis 30s key `dashboard:overview:<userRole>`.
**WHY:** 1 request = 1 round-trip TLS, 4 endpoint = 4× overhead. Payload < 20KB gzip. Cache per-role (admin thấy toàn bộ, planner thấy subset) = 2-4 cache entries, negligible.

### D39. "SKU hoạt động": query mỗi request hay materialized view?
**CHỐT:** Query mỗi request với Redis cache 30s. `SELECT COUNT(*) FROM master.item WHERE is_active=true AND deleted_at IS NULL` chạy < 5ms với index partial.
**WHY:** 10k items scan < 5ms với index, không cần MV refresh overhead. MV refresh 5m = 5m stale risk. Cache 30s Redis đủ perf cho dashboard auto-refresh 30s.

### D40. OrdersReadinessTable: top 5 hay pagination?
**CHỐT:** Top **10** hard-coded theo `due_date ASC, priority ASC`. Không pagination trên dashboard widget. Link "Xem tất cả" → `/orders?status=CONFIRMED,IN_PRODUCTION&sort=due_asc`.
**WHY:** Dashboard = quick scan, pagination phá flow. Top 10 = visible trên laptop 1366×768 không scroll. Link sang full list khi cần deep dive.

## §9 Test Strategy Execution

### D41. State machine test: table-driven hay describe/it?
**CHỐT:** **Table-driven** nested `for...for`: `for (from of ALL_STATES) for (to of ALL_STATES) it(`${from}→${to}`, ...)` = 100 test (plan §5.5 đã code đúng). Thêm 10 test riêng cho admin role override + 5 test edge case (invalid role, missing row, concurrent).
**WHY:** 100 test case tay viết = 100 copy-paste bug. Table-driven DRY, coverage đủ matrix. Edge case explicit vì logic khác matrix.

### D42. Recursive CTE benchmark: pg_stat trong test hay manual?
**CHỐT:** Benchmark file `bom-explode.bench.ts` (Vitest) với `console.time` + output JSON metrics. CI KHÔNG fail trên threshold (tránh flaky), chỉ warn. Manual `EXPLAIN ANALYZE` khi debug perf regression (không trong test suite).
**WHY:** CI perf test flaky trên 2 vCPU VPS → false positive tốn dev time. Log metrics vào artifact, review manual trend. EXPLAIN ANALYZE = dev tool, không auto.

### D43. Playwright fixture: seed script shared?
**CHỐT:** Fixture helper `e2e/fixtures/seed-cnc-abc.ts` export `seedCncAbcTemplate()` insert 1 template + R01 RELEASED + 25 component + 1 order SO-TEST + snapshot + 75 line. Reuse across 3 Playwright spec + unit test shortage. Cleanup `afterAll(() => truncateTestData())`.
**WHY:** DRY fixture = 1 source of truth BOM test. Nếu mỗi test seed riêng → drift + mất 30 phút mỗi khi BOM thay đổi spec.

## §10 Deploy + Migration 0003

### D44. Migration 0003 split: theo pattern 0002a/b/c?
**CHỐT:** 3 file:
- `0003a_schemas_and_types.sql` (superuser: CREATE SCHEMA engineering/sales/audit + GRANT USAGE).
- `0003b_bom_engineering.sql` (app user: bom_template, bom_revision, bom_component, eco_request).
- `0003c_sales_snapshot.sql` (app user: customer_order, order_line, bom_snapshot, snapshot_line, state_transition_log).
Apply script `./deploy/scripts/apply-sql-migrations.sh 0003a 0003b 0003c` sequential.
**WHY:** Plan §2.1 đã có 3 file. Split theo privilege (superuser vs app) giống 0002 pattern → script reuse. File đủ lớn (~200 LOC mỗi) nhưng 1 logical group dễ review.

### D45. Rollback migration fail giữa chừng?
**CHỐT:** 3-step recovery:
1. Pre-migration: `pg_dump -Fc iot > /backup/pre-0003-$(date +%s).dump` (2 phút).
2. Apply 0003a/b/c. Nếu fail giữa chừng → `DROP SCHEMA engineering CASCADE; DROP SCHEMA sales CASCADE; DROP SCHEMA audit CASCADE` (rollback 0003_rollback.sql).
3. Nếu corruption: `pg_restore -c -d iot /backup/pre-0003-*.dump` (5 phút).
Automation: script `./deploy/scripts/safe-migrate.sh 0003` wrap `pg_dump → apply → verify → rollback on fail`.
**WHY:** pg_dump Fc < 10MB cho DB hiện tại (empty tables V1.0) → backup nhanh. Verify sau apply = `\dt engineering.*` count 4 tables. Script giảm human error.

## §11 Scope Clarification (BLOCKING decisions)

### D46. ECO auto-apply khi approve: V1.1 defer?
**CHỐT:** **DEFER V1.3** (plan §13 D7). V1.1: ECO APPROVED → button "Apply" disabled + tooltip. API `/apply` return 501.
**WHY:** Auto-apply phá discipline bất biến snapshot (R2:245). Cần workflow: create new revision R02 từ ECO → release → chỉ apply cho order MỚI (old order giữ R01). Phức tạp 1 tuần dev.

### D47. Multi-level phantom 4+ level: V1.1 chỉ 3 level?
**CHỐT:** V1.1 support **tới depth 10** (CTE `WHERE depth < 10`), không limit 3 level. Phantom ở bất kỳ depth nào đều được explode transparent.
**WHY:** Plan §13 D1 nói "1 level phantom" nhưng CTE code (§6.2) support unlimited. Chốt code behavior thắng: 10 depth max, phantom pass-through không phân biệt depth. Edge case "phantom chứa phantom" hiếm, ok.

### D48. Reservation FIFO/FEFO: V1.1 chỉ mark?
**CHỐT:** **V1.1 defer hoàn toàn** reservation logic. `reserved_qty` field tồn tại trong schema nhưng KHÔNG có code nào SET > 0. V1.2 sẽ có `stock_reservation` entity + FIFO policy.
