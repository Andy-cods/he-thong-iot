# V1.3 Brainstorm — Production Loop: WO + Reservation + Assembly + ECO

**Ngày:** 2026-04-19
**Tag tham chiếu:** v1.2.0 LIVE tại https://mes.songchau.vn
**Scope:** Đóng nửa bên phải của snapshot state machine (AVAILABLE → RESERVED → ISSUED → ASSEMBLED → CLOSED), mở WO, mở ECO, khép lot/serial E2E, đặt móng QC in-process cho V1.4.
**Thời lượng dự kiến:** 14-18 ngày (1 dev) hoặc 8-12 ngày (2 agent parallel).
**Phong cách:** Brutal honesty. Không tô hồng. Gọi đúng tên debt, đúng tên risk.

---

## TL;DR (đọc 30 giây)

V1.3 **không phải** là 6 module độc lập. Nó là **1 pipeline sản xuất** (WO → Reservation → Scan → Issue → Assembly → Close) **cộng với 1 module độc lập** (ECO) **cộng với 1 lớp audit xuyên suốt** (Lot/Serial E2E) **cộng với 1 stub V1.4** (QC in-process).

Core CRUD module (WO, ECO, QC) là 1-3 ngày mỗi cái — nhàm chán.
**Core đau đầu duy nhất: Reservation policy FIFO/FEFO + deadlock khi 2 order grab cùng 1 lot.** Cái này sai là toàn bộ V1.3 sụp. Ưu tiên tuyệt đối.

**Nhỏ nhưng hay bị bỏ quên:**
- Assembly scan PWA phải reuse đúng pattern Receiving (offline queue + idempotency) — không tự phát minh.
- ECO apply với N affected orders: sync < 10, async BullMQ ≥ 10. **Đừng** vừa làm vừa chờ server timeout.
- WO **không** tự tạo khi confirm order. Manual trước, automation sau (V1.4+).

**Defer thẳng tay:** Multi-plant, serial genealogy cha-con, reservation expiry auto-release, partial WO split shift, QC plans per-item-category.

### Module 2: Reservation FIFO/FEFO — **3 ngày CORE**

Đây là module **khó nhất** và **quan trọng nhất** V1.3. Sai là cả pipeline sụp.

**Schema delta:**
```sql
CREATE TABLE app.reservation (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  snapshot_line_id uuid NOT NULL REFERENCES app.bom_snapshot_line(id),
  lot_serial_id uuid NOT NULL REFERENCES app.inventory_lot_serial(id),
  reserved_qty numeric(18,4) NOT NULL CHECK (reserved_qty > 0),
  reservation_reason app.reservation_reason NOT NULL DEFAULT 'AUTO_FIFO',
  wo_id uuid REFERENCES app.work_order(id),
  reserved_at timestamptz NOT NULL DEFAULT now(),
  reserved_by uuid REFERENCES app.user_account(id),
  released_at timestamptz,
  released_by uuid REFERENCES app.user_account(id),
  notes text
);
CREATE INDEX reservation_snapshot_idx ON app.reservation(snapshot_line_id);
CREATE INDEX reservation_lot_idx ON app.reservation(lot_serial_id);
CREATE INDEX reservation_active_idx ON app.reservation(snapshot_line_id) WHERE released_at IS NULL;
```

**Enum mới:** `reservation_reason` = AUTO_FIFO | AUTO_FEFO | MANUAL | OVERRIDE.

**Chính sách chọn lot (core logic):**
- **FIFO:** `ORDER BY lot.created_at ASC` (ngày nhận lô cũ nhất trước). Dùng cho item không perishable.
- **FEFO:** `ORDER BY lot.exp_date ASC NULLS LAST` (hết hạn gần nhất trước). Dùng cho item có `exp_date`.
- **Policy per-item:** cột `item.expiry_days` (đã có hoặc phải ADD vào `item`). Nếu `expiry_days > 0` → FEFO, else FIFO.
- **Kiểm tra lot eligible:** `status = 'AVAILABLE'` AND `lot.exp_date > now() + interval '7 days'` (không reserve lot sắp hết hạn < 7 ngày — buộc QC hold trước).

**API count: 4**
1. `POST /api/reservations` — auto reserve snapshot_line (input: `{snapshotLineId, qty, policy?}`).
2. `DELETE /api/reservations/[id]` — release (update `released_at`, UNRESERVE inv_txn).
3. `POST /api/reservations/manual` — manual override (admin chọn lot cụ thể, log reason).
4. `GET /api/reservations?snapshotLineId=xxx` — list reservations cho 1 snapshot_line.

**UI:**
- Button **"Reserve"** trong Snapshot Board ở state AVAILABLE. Click → auto FIFO/FEFO, modal xác nhận show lot nào được chọn, qty bao nhiêu, có muốn manual override không.
- Panel detail snapshot_line mở rộng hiển thị bảng reservation active.

**Performance & concurrency — đây là phần brutal:**

1. **Deadlock scenario:**
   - User A reserve snapshot_line_1 → cần qty 10 của item X, scan lot L1 (on_hand 15).
   - User B reserve snapshot_line_2 → cũng cần qty 10 của item X, cũng lot L1.
   - Cả 2 transaction đọc lot L1 on_hand = 15 thấy đủ → cả 2 INSERT reservation 10 → tổng 20 > 15 → **over-reservation**.

2. **Giải pháp V1.3:**
   - **Postgres advisory lock per (item_id)**: `SELECT pg_advisory_xact_lock(hashtext(item_id::text))` đầu transaction. Serialize theo item, không theo lot (ít deadlock cross-lot).
   - Timeout lock: `SET LOCAL lock_timeout = '5s'`. Nếu timeout → return 409 "ITEM_LOCKED_TRY_AGAIN".
   - Sau lock: re-query lot on_hand qty = sum(inv_txn) − sum(reservation active). Nếu không đủ → reject.

3. **Đừng:**
   - SELECT FOR UPDATE trên `inventory_lot_serial` row — dễ deadlock vì nhiều txn grab khác lot cùng item.
   - Dùng serializable isolation — performance kém, V1.3 read committed + advisory lock là đúng pattern.

**Test case bắt buộc (brutal):**
- Concurrent reserve 2 user cùng lot → 1 thành công, 1 reject với 409.
- Reserve qty > on_hand → reject.
- Reserve lot HOLD → reject.
- Release reservation → on_hand restore.
- FIFO: 3 lot ngày khác nhau → lot cũ nhất được pick.
- FEFO: 3 lot exp khác nhau → gần hạn nhất được pick.
- Mixed: item có expiry_days=30 → auto FEFO. Item không có → auto FIFO.

**Brutal:** Nếu không viết được 6 test case trên thì module này CHƯA XONG. Không chấp nhận "test sau".

### Module 3: Assembly Scan + Issue — **3 ngày**

**Schema delta:** Không cần ADD table mới (assembly_scan đã có). Chỉ cần:
- ALTER `assembly_scan` ADD COLUMN `wo_id uuid REFERENCES work_order(id)` (vì V1.3 scan-by-WO không scan-by-AO nữa).
- ALTER `assembly_scan` ADD COLUMN `lot_serial_id uuid REFERENCES inventory_lot_serial(id)` (scan barcode lot).

**Flow end-to-end:**
1. Operator mở `/pwa/assembly/[woId]` (WO đang IN_PROGRESS).
2. UI list component cần consume từ `work_order_line → snapshot_line` với `reserved_qty`.
3. Scan barcode = `lot_code` hoặc `serial_code`.
4. App lookup lot → check reservation match snapshot_line_id?
5. Nếu match → show qty còn lại cần consume, user confirm.
6. POST `/api/assembly/scan` với `{woId, snapshotLineId, lotSerialId, qty, offlineQueueId}`.
7. Server trong transaction:
   - Verify reservation active cho (snapshot_line_id, lot_serial_id) với qty >= requested.
   - INSERT `assembly_scan` (idempotent qua offline_queue_id).
   - INSERT `inventory_txn` kind=`ASSEMBLY_CONSUME`, negative qty.
   - UPDATE `reservation.released_at = now()` nếu consume hết reserved_qty.
   - UPDATE `bom_snapshot_line.issued_qty += qty`, `assembled_qty += qty` (V1.3 không tách issue/assemble, scan = cả 2).
   - Nếu `assembled_qty >= required_qty` → auto transition RESERVED → ASSEMBLED.
   - UPDATE `lot_serial.status = 'CONSUMED'` nếu on_hand = 0.

**API count: 3**
1. `POST /api/assembly/scan` — main endpoint, idempotent qua offlineQueueId.
2. `POST /api/assembly/wo/[id]/complete` — mark WO COMPLETED, snapshot_lines → CLOSED.
3. `GET /api/assembly/wo/[id]/components` — component list cho scan screen.

**UI: 1 page** `/pwa/assembly/[woId]`
- Reuse component pattern từ Receiving PWA (đã có offline queue Dexie + background sync).
- Top: WO header + progress %.
- Middle: list component row (SKU, required_qty, issued_qty, status, reserved lot).
- Bottom: scan input (barcode scanner hoặc manual type).
- Toast: "Scan OK: lot L-240419-001 qty 1/10" hoặc "Lot không khớp reservation".

**Worker queue:**
- Thêm queue `assembly-scan-sync` trong worker (đã có placeholder trong `apps/worker/src/index.ts`).
- Job: xử lý batch scan từ offline queue PWA khi sync → gọi cùng logic /api/assembly/scan nhưng idempotent.

**Risk:**
- **Offline scan sync conflict:** 2 device scan cùng lot cùng qty offline, sync lúc online → idempotency key chỉ check offline_queue_id. Nếu 2 device generate khác id nhưng cùng lot → over-consume. **Giải pháp:** advisory lock per lot khi insert scan.
- **Scan sai snapshot_line:** operator scan lot không reserve cho WO này → reject rõ ràng "Lot reserved cho WO khác". Không tự động re-route.

**Brutal:** Assembly PWA nhìn đơn giản, thực tế là module test nhiều nhất. Reuse Receiving pattern tuyệt đối, đừng viết lại IndexedDB/sync layer.

### Module 4: ECO Apply Workflow — **2 ngày**

**Schema delta:**
```sql
CREATE TYPE app.eco_status AS ENUM (
  'DRAFT', 'SUBMITTED', 'APPROVED', 'APPLIED', 'REJECTED'
);

CREATE TABLE app.eco_change (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  eco_no varchar(64) NOT NULL UNIQUE, -- ECO-YYMM-####
  title varchar(256) NOT NULL,
  description text,
  affected_template_id uuid NOT NULL REFERENCES app.bom_template(id),
  affected_revision_id uuid NOT NULL REFERENCES app.bom_revision(id),
  new_revision_id uuid REFERENCES app.bom_revision(id), -- null cho đến khi APPROVED
  status app.eco_status NOT NULL DEFAULT 'DRAFT',
  submitted_at timestamptz,
  submitted_by uuid REFERENCES app.user_account(id),
  approved_at timestamptz,
  approved_by uuid REFERENCES app.user_account(id),
  applied_at timestamptz,
  applied_by uuid REFERENCES app.user_account(id),
  affected_orders_count int DEFAULT 0,
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE app.eco_line (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  eco_id uuid NOT NULL REFERENCES app.eco_change(id) ON DELETE CASCADE,
  change_kind varchar(16) NOT NULL, -- ADD | REMOVE | QTY_CHANGE | REPLACE
  bom_line_id uuid REFERENCES app.bom_line(id), -- nullable for ADD
  new_component_item_id uuid REFERENCES app.item(id),
  new_qty_per_parent numeric(18,6),
  reason text
);
```

**Flow ECO:**
1. User tạo ECO DRAFT → chọn template + revision cần thay đổi, thêm N eco_line (delta).
2. Submit → DRAFT → SUBMITTED (notification cho Approver).
3. Approve → SUBMITTED → APPROVED → **auto clone revision**:
   - INSERT `bom_revision` mới với `version = old_version + 1`, status DRAFT.
   - Apply delta vào `bom_line` children của new revision.
   - Set `eco_change.new_revision_id`.
4. Apply → APPROVED → APPLIED:
   - RELEASE new revision (`bom_revision.status = RELEASED`, old revision SUPERSEDED).
   - Count `affected_orders = SELECT COUNT(*) FROM sales_order WHERE revision_id = old_revision_id AND state NOT IN ('CLOSED')`.
   - Nếu `affected_orders < 10`: sync warn UI, gắn flag `orders.requires_review = true`.
   - Nếu `>= 10`: enqueue BullMQ job `eco-apply-batch` → background process warn từng order.
5. Reject → SUBMITTED → REJECTED (audit log reason).

**API count: 6**
1. `POST /api/eco` — create draft.
2. `GET /api/eco` — list.
3. `GET /api/eco/[id]` — detail.
4. `POST /api/eco/[id]/submit` — DRAFT → SUBMITTED.
5. `POST /api/eco/[id]/approve` — SUBMITTED → APPROVED + clone revision.
6. `POST /api/eco/[id]/apply` — APPROVED → APPLIED + supersede + affected orders.

(Reject = POST `/api/eco/[id]/reject` gộp với approve endpoint bằng body `{decision: 'reject', reason}`.)

**UI: 3 screens**
- `/eco` — list + filter status.
- `/eco/new` — form create với multi-line delta.
- `/eco/[id]` — detail + approve/apply buttons + affected orders section.

**Risk:**
- **Delta apply sai bom_line:** nếu user chọn `bom_line_id` của revision cũ nhưng clone revision sinh bom_line_id mới → phải map theo `component_item_id + parent_line_id` khi apply.
- **Revision version conflict:** 2 ECO APPROVED đồng thời cho cùng template → chỉ 1 apply thành công, ECO còn lại phải re-clone từ revision mới. **V1.3 giản lược:** lock per template_id, 1 ECO APPLIED tại 1 thời điểm.
- **Affected orders warn async:** nếu BullMQ worker down → warn không tới user. Lưu `eco_change.applied_at` khi enqueue success, show progress % ở UI.

**Brutal:** ECO là feature flashy, nhưng V1.3 dùng rất ít (< 5 ECO/tháng kỳ vọng). Đừng tốn quá 2 ngày. Giao diện đẹp cho V1.4.

### Module 5: Lot + Serial Tracking E2E — **2 ngày**

**Schema delta:** Không cần. `inventory_lot_serial` + `inventory_txn.lot_serial_id` đủ.

**Mục tiêu:** 1 page `/lot-serial/[id]` hiển thị **toàn bộ lifecycle** của 1 lot/serial:
- Received: từ PO nào, supplier nào, receipt khi nào, qty ban đầu.
- Reserved for: list reservations active/historical với WO/snapshot_line.
- Issued to: list assembly_scan consume.
- Consumed at: inventory_txn ASSEMBLY_CONSUME + final on_hand.
- QC: lot status AVAILABLE/HOLD/CONSUMED/EXPIRED + lịch sử thay đổi.

**Query strategy:**
- Option A: runtime query JOIN 4 bảng (inv_txn, reservation, assembly_scan, receipt) mỗi lần user mở page. **Đủ nhanh V1.3** (1 lot có max ~20 related rows).
- Option B: materialized view `lot_history` refresh hourly. **Over-engineer cho V1.3.** Defer V1.4 khi có > 1000 lot/day.

**Chọn option A.** Đơn giản, đúng KISS.

**API count: 1**
- `GET /api/lot-serial/[id]` — full detail + history array.

**UI: 1 page** `/lot-serial/[id]`
- Header: lot_code, item SKU, supplier, mfg_date, exp_date, status badge.
- Timeline section (vertical timeline pattern):
  - 19/04/2026 10:30 — Received (GRN-XXX, qty 100) [link]
  - 19/04/2026 14:00 — Reserved qty 10 cho SO-2604-0001 / line X [link]
  - 20/04/2026 09:15 — Issued qty 10 cho WO-2604-0003 [link]
  - 20/04/2026 09:20 — Consumed qty 10, on_hand = 0
- Footer: current on_hand calculation breakdown.

**Brutal:** Đây là "feature trang trí" — hay show khách hàng nhưng operator hiếm mở. Đừng over-design. 2 ngày là max.

## §7. Performance concerns — brutal breakdown

### 7.1 Reservation deadlock (CRITICAL)

- **Threat:** 2-5 operator reserve concurrent → over-reservation.
- **Mitigation:** `pg_advisory_xact_lock(hashtext(item_id))` + `lock_timeout = 5s`.
- **Test target:** 10 concurrent request cùng item, chỉ 1 thành công/tick, rest retry hoặc 409.

### 7.2 ECO apply cascading (MEDIUM)

- **Threat:** ECO APPLY với 50 affected orders → block request 30s+.
- **Mitigation:** threshold 10 orders — dưới sync, trên enqueue BullMQ `eco-apply-batch`.
- **UI:** progress bar polling `/api/eco/[id]/progress` every 5s khi async.

### 7.3 Lot history query (LOW)

- **Threat:** page `/lot-serial/[id]` JOIN 4 bảng.
- **Reality:** 1 lot có max 30-50 related rows (receipt + N reservation + N scan + N txn). Query < 100ms.
- **Mitigation:** None needed V1.3. Materialized view defer V1.4.

### 7.4 Assembly scan concurrent (MEDIUM)

- **Threat:** 5 tablet scan cùng lúc cùng WO → race condition reservation release.
- **Mitigation:** advisory lock per lot_id trong endpoint `/api/assembly/scan`. Với 5 tablet cùng lot (hiếm): 1 thành công, rest retry hoặc report "lot đã consume".
- **Queue-based alternative:** defer V1.4 nếu volume > 100 scan/min.

### 7.5 WO list với snapshot_lines eager load (LOW)

- **Threat:** `/work-orders` list 100 WO × 20 line/WO = 2000 row query.
- **Mitigation:** lazy load lines chỉ khi expand. List chỉ show summary (count lines, good_qty/planned_qty).

### 7.6 Snapshot Board "Reserve" button với lookup lot (MEDIUM)

- **Threat:** mỗi click "Reserve" query FIFO/FEFO trên `inventory_lot_serial` có thể slow nếu > 10k lot.
- **Mitigation:** partial index `lot_available_idx ON inventory_lot_serial (item_id, exp_date, created_at) WHERE status = 'AVAILABLE'`. Đã trong 0006c.
