## 1. Goal + success criteria

### 1.1. Trạng thái kết thúc V1.7-beta

User click BOM từ `/bom` → landing thẳng `/bom/[id]/grid` (đã có từ V1.7-alpha). Tại đây thấy:

- **Topbar h-12** hiển thị BOM code + name + status badge + KPI chips (Đơn hàng · WO · Thiếu · ECO) + menu actions, thay vì ContextualSidebar 220px bên trái.
- **Global sidebar 220px** giữ full-label (bỏ icon-only 56px mode cũ V1.6).
- **Grid** chiếm khoảng 65-70% viewport (trên bottom panel, dưới topbar).
- **Bottom Panel** h-9 tab bar + content collapsible (default 280px, resize 120-60vh). 6 tab: Đơn hàng / Lệnh SX / Mua sắm / Thiếu hàng / ECO / Lắp ráp.
- **Click tab** (hoặc click KPI chip Topbar) → panel expand, hiển thị list filter theo `bomTemplateId`.
- **Edit cell "Loại"** (col 3) qua dropdown → persist `bom_line.metadata.kind` JSONB, badge `⚠` hiện khi override `item.itemType`.
- **Double-click cell "Loại"** (hoặc icon `⋯`) → Side Sheet phải 480px với tab Thương mại/Gia công, form Tier 1 (NCC + lead time + MOQ + giá cho com; vật liệu + kích thước phôi + ghi chú route cho fab). Persist `metadata.sourcing` / `metadata.routing`.
- **Button "Lịch sử"** Topbar → right drawer 440px timeline activity log.
- **Link cũ** `/bom/[id]/orders` v.v. vẫn hoạt động (307 redirect tới grid với panel auto-open).

## 6. Risks + mitigations

### R1 — ContextualSidebar removal break existing flow

**Risk:** Sidebar 220px cũ có 9 nav link; có code ngoài layout import nó không?

**Mitigation:** Commit 2 — chạy `grep -r "ContextualSidebar" apps/` trước khi DELETE, confirm chỉ `layout.tsx` import. Cũng grep `buildBomContextNav` — nếu còn consumer khác thì giữ function thay vì xoá.

**Fallback:** Nếu phát hiện consumer — comment out function + TODO migrate sang Topbar props thay vì DELETE file.

### R2 — URL state bottom panel conflict với existing query params

**Risk:** `?page=` của pagination, `?q=` của search — conflict namespace với `?panel=`, `?collapsed=`, `?h=`, `?autoOpen=`, `?drawer=`.

**Mitigation:** Prefix `bp_` cho panel state (`?bp_panel=orders&bp_h=280&bp_collapsed=0`). `drawer` riêng cho right drawer. `autoOpen` 1-shot flag (grid page consume rồi strip). Grid page không dùng query khác nên an toàn; nhưng các Panel list (OrdersPanel) khi được render trong bottom panel **không** dùng `?page=` / `?q=` từ URL (chúng dùng local state hoặc nested ?bp_q=).

**Verify:** Test click tab, check URL không mất state khác.

### R3 — Univer `IDataValidation` list API compat

**Risk:** Version Univer 0.21.0 trong project có hỗ trợ `IDataValidation` list rule không? Docs Univer ít.

**Mitigation:**
- Bước 1 commit 7: viết proof-of-concept 30 dòng chạy local — nếu API work → tiếp tục.
- Fallback 1: Shadcn Select overlay DOM — hit-test tọa độ cell col 3 via Univer `ICommandService` → portal Select tại vị trí cell. Effort +4h, bundle không thay đổi (Select đã có).
- Fallback 2: dùng cell style condition — cell col 3 render button mini "fab ▾" mở Popover inline. Kém smooth hơn nhưng work 100%.
- Quyết định budget: 2h POC. Nếu fail → chuyển Fallback 1.

### R4 — Persist kind/sourcing/routing conflict với Excel import flow

**Risk:** Import Excel hiện tại có ghi đè `metadata` toàn bộ không? Nếu có → mất kind user đã đặt.

**Mitigation:**
- Grep `metadata` trong `apps/web/src/server/services/excelImport*.ts` hoặc `apps/web/src/app/api/bom/templates/[id]/import*`.
- Nếu overwrite: sửa import logic → chỉ merge `{ size, seq, supplierItemCode }`, giữ `{ kind, sourcing, routing }` từ DB.
- Nếu chỉ patch fields cụ thể: an toàn, note trong WORKLOG.

**Verify commit 7:** sau khi persist kind test, thử re-import Excel cùng BOM → kind không mất.

### R5 — Bottom panel data fetch performance khi BOM có > 100 orders

**Risk:** `useOrdersList({ bomTemplateId, pageSize: 50 })` fetch mỗi lần user click tab. BOM nhiều đơn → 50 rows render + virtualize nhưng initial fetch blocks UI.

**Mitigation:**
- Lazy: chỉ fetch khi `activeTab === 'orders'` và panel expanded (conditional `enabled` React Query).
- Prefetch KPI count chỉ lấy từ `useBomWorkspaceSummary` (đã có, rẻ — COUNT query).
- Skeleton 3 rows khi loading.

### R6 — Redirect loop sub-route

**Risk:** Nếu `/bom/[id]/grid` cũng bị catch bởi redirect logic → infinite loop.

**Mitigation:** Redirect chỉ áp cho 7 đường dẫn hardcoded. `/grid` không trong list. Test curl `-I` từng path.

### R7 — Smoke test CI chạy lâu do thêm assertion

**Risk:** 45 assertion × curl → CI 30s → 50s.

**Mitigation:** Acceptable. Nếu tăng > 2min → optimize parallel curl `xargs -P 4`.

- Commit tuần tự commit 1 → commit 9 theo section 3.

- `.github/workflows/ci.yml` phải PASS: typecheck + build + smoke test (cần setup smoke test trong CI bằng docker-compose ephemeral hoặc skip nếu không có — ít nhất typecheck + build).
- Manual smoke test local hoặc staging VPS trước merge main.

### 7.3. Deploy to production

- Merge `main` → GitHub Actions `.github/workflows/deploy.yml` tự build image `ghcr.io/andy-cods/hethong-iot:latest` → SSH VPS `docker pull + docker compose up -d app worker caddy`.
- Chạy `bash tests/smoke/v1.7-smoke.sh https://mes.songchau.vn` — target ≥ 42 PASS.
- Test thủ công flow 5.2 manual test trên production.

### 7.4. Rollback plan

Nếu deploy fail:
1. **Option A (preferred):** `git revert <merge-commit> && git push main` → CI rebuild với mã cũ → deploy.
2. **Option B (nhanh):** SSH VPS `docker tag ghcr.io/andy-cods/hethong-iot:v1.7.0-alpha ghcr.io/andy-cods/hethong-iot:latest && docker compose up -d app worker`.
3. Nếu chỉ UI bug → hotfix commit nhỏ trên main, không revert.

Data: persist metadata.kind/sourcing/routing **không gây data loss** vì chỉ append JSONB key. Rollback code không xoá data đã ghi; field metadata mới nằm yên.

### 7.5. Feature flag (không dùng)

YAGNI — V1.7-beta không cần feature flag. User production = admin + 2-3 planner nội bộ; rollback nhanh hơn FF.
