# V1.4 Brainstorm — Security Hardening & Operational Readiness

**Ngày:** 2026-04-19
**Tag tham chiếu:** v1.3.0 LIVE (WO + Reservation + Assembly + ECO + Lot/Serial E2E + QC stub)
**Scope:** Đóng vết thương bảo mật còn hở (RLS, RBAC matrix đầy đủ, rate limit, session mgmt), hoàn thiện Audit như công cụ vận hành thật, và dựng stack Observability/Backup/Load-test đủ để gọi hệ thống là "production-grade" chứ không phải "chạy được".
**Thời lượng dự kiến:** 12-16 ngày (1 dev) hoặc 7-10 ngày (2 agent parallel).
**Phong cách:** Brutal honesty — không tô hồng, gọi đúng tên debt và risk.

---

## TL;DR (đọc 30 giây)

V1.4 **không phải** là 9 module độc lập. Nó là **3 block** ghép lại:

1. **Block Security (P0):** RLS + RBAC 12×12 + Rate limit. Đây là 3 lớp guard bắt buộc phải làm CÙNG NHAU, không tách được — vì RBAC app-layer mà không có RLS database-layer thì chỉ cần 1 dev quên `where userId=...` là lộ data; và RLS mà không có RBAC app thì UI rối loạn 403.
2. **Block Operational (P1):** Audit polish + Session mgmt + Reset password. Đây là 3 UI cho admin — phải làm cùng `/admin/settings` hub để tránh UX chắp vá.
3. **Block Infra (P2):** Monitoring + Backup cron + Load test. Đây là work dưới gầm xe — ít user thấy, nhưng nếu thiếu thì đêm VPS crash lúc 3h sáng không ai biết.

**Core đau đầu duy nhất: RLS strict mode breakage.** Bật RLS là mặc định deny — nếu quên 1 policy nào đó, endpoint đang chạy OK sẽ đột nhiên 500 trên prod. Cái này sai là toàn bộ V1.4 "LIVE" trong nửa giờ rồi rollback. Ưu tiên tuyệt đối: test RLS dưới cả 4 role × 12 entity trước khi deploy.

**Nhỏ nhưng hay bị bỏ quên:**
- RBAC matrix hiện tại là ad-hoc `hasRole()` rải rác trong ~30+ file — cần **1 file duy nhất** `permissions.ts` với bảng, cộng với **middleware** gọi bảng đó. Đừng fix từng file.
- Session revoke UI rất dễ, nhưng revoke **immediate** (tức là JWT access token đang trong tay user cũng mất hiệu lực trong < 5 phút) đòi hỏi session check mỗi request → cost. Decision: revoke = mark `revoked_at`, JWT access token hết hạn sau 15 phút tự nhiên. V1.5+ mới làm Redis blacklist nếu thật cần.
- Rate limit login 5/min/IP nghe đơn giản, nhưng IP sau Cloudflare là `CF-Connecting-IP` header chứ không phải `req.ip` → check middleware đọc đúng header, không thì toàn bộ rate limit fail (1 IP Cloudflare = toàn internet).
- Load test 100 concurrent **không có ý nghĩa** nếu schema query không đi qua index — phải chạy `EXPLAIN ANALYZE` trước. Load test là bài test cuối, không phải bài test phát hiện N+1.

**Defer thẳng tay:** SSO/OAuth (Google/Azure), MFA (TOTP/WebAuthn), audit forensics tool, row-level encryption, data masking, SOC2 compliance artifacts, penetration test thuê ngoài.

## §1. 9 module V1.4 breakdown — effort thực tế

### Module 1: RLS Policies — **3 ngày** — P0 Security critical

**Scope:**
- Bật RLS cho **10 bảng nghiệp vụ chính**: `item`, `bom_template`, `bom_revision`, `sales_order`, `bom_snapshot`, `purchase_request`, `purchase_order`, `work_order`, `assembly_order`, `audit_event`.
- **KHÔNG bật** cho: `role`, `user_account`, `session` (admin-only, guard app-layer đủ), `inventory_txn` (append-only, read scope theo item → gắn RLS qua view sẽ phức tạp, defer V1.5).
- Mỗi bảng có 2-4 policy: SELECT/INSERT/UPDATE/DELETE per role.

**Pattern:**
```sql
ALTER TABLE app.item ENABLE ROW LEVEL SECURITY;
ALTER TABLE app.item FORCE ROW LEVEL SECURITY;  -- chặn cả owner bypass

-- admin: full access
CREATE POLICY item_admin_all ON app.item
  FOR ALL TO app_admin USING (true) WITH CHECK (true);

-- planner/warehouse/operator: read-only
CREATE POLICY item_read_all ON app.item
  FOR SELECT TO app_planner, app_warehouse, app_operator
  USING (true);

-- planner: create/update master data
CREATE POLICY item_planner_write ON app.item
  FOR INSERT TO app_planner
  WITH CHECK (true);

CREATE POLICY item_planner_update ON app.item
  FOR UPDATE TO app_planner
  USING (true) WITH CHECK (true);

-- delete: chỉ admin (không có policy cho role khác → deny)
```

**Kết nối DB per-request:**
- `SET LOCAL role = 'app_planner'` đầu transaction, dựa vào `session.roles[0]` (primary role).
- Nếu multi-role (admin,planner) → ưu tiên quyền cao nhất = admin.
- Connection pool dùng superuser `iot_app` (BYPASSRLS=false), `SET ROLE` per-request.

**Acceptance:**
- 4 role × 10 entity = **40 smoke test** pass (matrix trong §4).
- Smoke test kiểu: login as operator, call `POST /api/items` → 403 RLS (không phải 200).
- Tắt RLS trên 1 bảng (simulate bug) → test fail ngay → bật lại.

**Risk cao nhất:** Sở hữu entity (ownership-based) trong V1.4 chưa áp dụng vì rẻ. Ví dụ: "planner chỉ thấy order mình tạo" — **defer V1.5**. V1.4: role-based SELECT scope = all rows trong table.

**Brutal:** Đừng cố "clean" RLS từ đầu với ownership. Làm role-based trước, đúng 4 role, xong. V1.5 mới add `created_by_user_id` filter nếu cần. Nếu làm hết cả 2 trong 1 sprint → kéo 7 ngày.

### Module 2: RBAC 12×12 Matrix — **2 ngày** — P0 Security critical

**Scope:**
- Thay thế `role-guard.ts` `hasRole()` ad-hoc bằng **1 bảng permission** + **1 helper** `can(user, action, entity)`.
- Matrix **4 role × 12 entity × 6 action** = 288 cell, hardcode trong 1 TS file.
- Middleware `apps/web/src/server/permissions.ts` — gọi trong mọi API route thay cho `requireRole()`.

**File mới:**
- `apps/web/src/server/permissions.ts` — bảng + helper.
- `packages/shared/src/rbac/matrix.ts` — share giữa FE và BE để hide/show button UI.

**Bảng tham khảo:** §4.

**API delta:**
- Không thêm route mới. Refactor 30+ route hiện tại để gọi `can(session, 'create', 'item')` thay `requireRole('admin')`.
- UI: thay `{isAdmin(user) && <Button>Delete</Button>}` bằng `{can(user, 'delete', 'item') && <Button>Delete</Button>}`.

**Acceptance:**
- Matrix diff test: `tests/rbac-matrix.test.ts` iterate 288 cell, assert expected outcome so với `plans/v1.4/rbac-reference.md`.
- Không còn `hasRole(...)` raw trong `src/app/api/**/route.ts` (eslint rule ban).

**Brutal:** Matrix 12×12 viết ra là DỄ. Cái khó là **đi refactor 30+ route** dùng lại helper. Nếu skip refactor → RBAC matrix là phế liệu TypeScript, production vẫn dùng `hasRole('admin')` cũ. Effort 2 ngày là gồm **refactor thật**, không chỉ viết file mới.

### Module 3: Audit UI Enhance — **2 ngày** — P1 Operational

**Scope:**
- `/admin/audit` hiện tại (V1.2) chỉ list + filter actor/action. V1.4 thêm:
  1. **Diff viewer** side-by-side: `beforeJson` ↔ `afterJson` với highlight field thay đổi.
  2. **Rollback stub** (UI only, không execute thật) — button "Proposed rollback SQL" show statement `UPDATE ... SET ... WHERE id=...` dựa trên `beforeJson`. Copy to clipboard. Admin chạy thủ công.
  3. **Export Excel**: filter current → download `.xlsx` với cột flattened (actor, action, entity, before, after, notes, occurred_at).
  4. **Linked object navigation**: click `objectId` trong row → deep-link về detail page của entity đó.

**API:**
- `GET /api/admin/audit` — đã có (pagination + filter). ADD filter `date_from`, `date_to`, `object_id`.
- `GET /api/admin/audit/export.xlsx` — stream Excel buffer từ ExcelJS (reuse pattern Import).

**UI:**
- Component `AuditDiffViewer` dùng `react-diff-viewer-continued` (npm, ~30KB). Nếu lazy-load.
- Action bar trong filter row: [Export Excel], [Clear filters].

**Acceptance:**
- Audit 1k event, filter by actor → export < 5s, file < 2MB.
- Diff viewer với `beforeJson = null` (CREATE) → hiển thị "Created" badge, skip diff.
- Rollback SQL preview đúng format (không execute, chỉ COPY).

**Brutal:** Rollback tự động là cạm bẫy. CASCADE, FK, trigger — 1 UPDATE sai là hỏng 10 bảng. V1.4 stub preview SQL đủ, admin đọc kỹ rồi paste vào `psql` chạy tay. V2+ mới làm executable rollback với approval workflow.

### Module 4: Session Management UI — **1 ngày** — P1 Operational

**Scope:**
- Trang `/admin/settings/sessions` — list active session của tất cả user (admin view) hoặc session của chính mình (user view).
- Cột: username, ip_address, user_agent, issued_at, expires_at, last_seen (nếu có), [Revoke].
- Click Revoke → `UPDATE session SET revoked_at = now() WHERE id = ?` → user đó tại lần refresh token tiếp theo sẽ bị đá về login.

**Schema delta (migration 0007b):**
```sql
ALTER TABLE app.session
  ADD COLUMN IF NOT EXISTS last_seen_at timestamptz,
  ADD COLUMN IF NOT EXISTS device_label varchar(128);
```

**API:**
- `GET /api/admin/sessions` — list all active (admin) hoặc filter by userId (self).
- `DELETE /api/admin/sessions/[id]` — revoke 1 session.
- `DELETE /api/admin/sessions/user/[userId]` — revoke all của 1 user (force logout).

**Middleware update:**
- `auth.ts` refresh token endpoint check `session.revoked_at IS NULL` — nếu revoked → 401, xóa cookie.

**Acceptance:**
- Revoke session A → dùng access token cũ vẫn OK tới khi JWT expire (15 phút). Sau đó refresh → 401. **Đây là expected behavior V1.4**, document trong CLAUDE.md.
- Revoke all cho user B → cả 3 browser tab đều logout trong 15 phút.

**Brutal:** "Revoke ngay tức khắc" = Redis session blacklist + check mỗi request. Cost 1 Redis roundtrip × 10k request/ngày = không đáng V1.4. Giữ 15 phút lag, user OK với SLA đó. V1.5+ đánh giá lại.

### Module 5: Reset Password Workflow — **1 ngày** — P1 Operational

**Scope:**
- **Admin reset user password** (primary case V1.4):
  1. `/admin/users/[id]` → button "Reset password".
  2. Modal confirm → BE gen 12-char temp password, `UPDATE user_account SET password_hash = hash(temp), must_change_password = true`.
  3. Admin thấy temp password **1 lần duy nhất** trong modal (không lưu DB plain). Copy để gửi user qua Telegram/SMS.
- **User self-reset via email** (V1.4 **stub**): form `/forgot-password` nhập email → luôn hiển thị "Nếu email tồn tại, link đã gửi" (ngăn user enumeration). Thực tế: log link vào `logger.info` (không gửi email thật V1.4).
- **Force change password**: nếu `must_change_password = true`, sau login redirect về `/settings/change-password` (không cho vào app). Update xong → `must_change_password = false`.

**Schema delta:**
```sql
ALTER TABLE app.user_account
  ADD COLUMN IF NOT EXISTS must_change_password boolean NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS password_reset_token_hash text,
  ADD COLUMN IF NOT EXISTS password_reset_expires_at timestamptz;
```

**API:**
- `POST /api/admin/users/[id]/reset-password` — admin reset, returns temp password 1 lần.
- `POST /api/auth/forgot-password` — user request reset, log link (stub).
- `POST /api/auth/reset-password` — user submit token + new password.

**Acceptance:**
- Admin reset user B → user B login bằng password cũ fail, login bằng temp password OK, sau login redirect force change.
- User request reset email không tồn tại → response giống hệt email tồn tại (user enumeration defense).

**Brutal:** Email service thật (SMTP, SendGrid, Mailgun) = tốn tiền + config DKIM/SPF + rate limit bên provider. V1.4 chỉ VPS nội bộ, workshop Việt Nam, admin dùng Telegram gửi temp password nhanh hơn. V1.5+ mới rigging SMTP khi có user > 50.

### Module 6: Rate Limit — **1 ngày** — P0 Security critical

**Scope:**
- **Login**: 5 request/phút/IP (username-agnostic — rate limit cả username fail lẫn success).
- **API burst**: 100 request/phút/IP cho tất cả route (except `/api/health`, `/api/ready`).
- **Password reset**: 3 request/giờ/IP.
- Backend: Redis INCR + EXPIRE.
- Response 429 kèm `Retry-After` header.

**Schema delta (migration 0007b):**
```sql
-- Fallback nếu Redis down (optional, log only)
CREATE TABLE app.rate_limit_log (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  bucket_key varchar(256) NOT NULL,
  violated_at timestamptz NOT NULL DEFAULT now(),
  ip_address varchar(64),
  user_agent text
);
CREATE INDEX rate_limit_log_key_idx ON app.rate_limit_log (bucket_key, violated_at DESC);
```

**Implementation:**
- Next.js middleware `apps/web/src/middleware.ts` — read IP từ `CF-Connecting-IP` (Cloudflare) hoặc `X-Forwarded-For` (Caddy) hoặc `req.ip`.
- Key format: `rl:{bucket}:{ip}` → TTL 60s.
- Lib: đừng dùng `@upstash/ratelimit` (cloud-only) — tự viết ~30 dòng với `ioredis`.

**Acceptance:**
- Curl login 6 lần liên tiếp cùng IP → lần 6 nhận 429 với `Retry-After: 45`.
- 100 request `/api/items` trong 1 phút → request 101 nhận 429.
- Health check spam không bị limit.

**Risk:** IP thật sau Cloudflare Tunnel là `CF-Connecting-IP`. Nếu đọc sai → toàn Cloudflare IP pool share 1 bucket → rate limit fire sau 5 user đăng nhập. **Test trên VPS thật có tunnel**, đừng chỉ test local.

**Brutal:** Đừng implement sliding window phức tạp. Fixed window 60s đủ cho V1.4. Lý do: workshop nhỏ, attacker tồi, không cần chống burst sophisticate.

### Module 7: Monitoring Stack (Prometheus + Grafana + Loki) — **2-3 ngày** — P2 Infra

**Scope:**
- **Compose add 3 service**: `prometheus` (1.5GB RAM), `grafana` (500MB), `loki` (1GB).
- Nhưng VPS chỉ 2GB RAM, đang dùng 1.4GB (Postgres + Redis + app + worker) → **không đủ chỗ chạy 3 container thêm** (~3GB).
- **Decision**: **V1.4 chỉ Prometheus + Grafana**, Loki defer V1.5 khi nâng VPS 4GB. Hoặc dùng **Grafana Cloud Free tier** (10k metrics + 50GB logs free) → không tốn RAM VPS.

**Option A: Self-hosted Prom+Grafana (RAM ~1.5GB thêm)** — cần nâng VPS 4GB trước. Cost: ~$10/tháng upgrade.

**Option B: Grafana Cloud Free tier** — Prometheus exporter chạy local, push lên Grafana Cloud. Cost: $0 đến 10k metrics.

**Recommendation:** **Option B** cho V1.4 (nhanh, không đụng RAM), upgrade VPS sau.

**Metrics cần export:**
- HTTP metrics: request count + duration histogram per route.
- DB connection pool: active/idle.
- BullMQ job: queue depth + processed + failed.
- Business metrics: snapshot_line count by state, reservation count, WO count by status.

**Alert rules:**
- 5xx rate > 1% trong 5 phút.
- DB connection pool full > 2 phút.
- BullMQ failed rate > 5%.
- Disk usage > 80%.

**Acceptance:**
- Grafana dashboard 4 panel: HTTP traffic, DB connections, Queue depth, Business state.
- 1 alert fire test: force 500 response 10 lần → alert vào Telegram bot < 5 phút.

**Brutal:** Setup dashboard đẹp tốn 2-3 ngày trivially. Đừng cố custom panel, import template Grafana community cho Next.js/Postgres/BullMQ (có sẵn). 80% dashboard là copy-paste.

### Module 8: Backup Cron Automation + S3/R2 Off-site — **1 ngày** — P2 Infra

**Scope:**
- Hiện tại: `deploy/scripts/backup.sh` có sẵn từ V1 (pg_dump | gpg → local file). Cron chưa setup.
- V1.4:
  1. Crontab: mỗi 02:00 sáng chạy backup → `/opt/hethong-iot/backups/YYYY-MM-DD.sql.gpg`.
  2. Upload S3/R2 off-site: `rclone copy` lên Cloudflare R2 (free 10GB storage + 10GB egress).
  3. Retention: local 7 ngày, R2 30 ngày.
  4. Telegram alert nếu backup fail hoặc size < 10MB (bất thường).
- Restore drill: document 1 file `deploy/docs/restore-drill.md` — 5 bước test restore vào container tạm.

**Script updates:**
- `deploy/scripts/backup.sh`:
  - Add `rclone copy "$OUT" r2:iot-backups/` sau pg_dump.
  - Add Telegram notify khi fail.
- `deploy/scripts/restore-drill.sh`: spin Postgres container, restore latest dump, verify count > 0.

**Secrets:**
- R2 credential trong `/opt/hethong-iot/secrets/.rclone.conf` (600 perm).

**Acceptance:**
- Cron chạy 1 đêm → next day check R2 có file.
- Restore drill lúc 03:00 Chủ Nhật weekly, log vào Telegram.
- Simulate backup fail (rename pg_dump) → Telegram alert < 1 phút.

**Brutal:** Tạo R2 account + rclone config tốn 30 phút. Đừng overcomplicate với multi-region, versioning enterprise. Free tier R2 đủ workshop 1 năm.

### Module 9: Load Test — **1-2 ngày** — P2 Infra

**Scope:**
- Seed 10k SKU (đã có template import V1.1).
- k6 script 100 VU × 5 phút cho 5 endpoint nóng:
  1. `GET /api/items?q=search` (pg_trgm query).
  2. `GET /api/snapshots/[id]` (join BOM snapshot_line).
  3. `POST /api/reservations` (advisory lock, concurrency).
  4. `GET /api/work-orders?status=IN_PROGRESS` (index scan).
  5. `POST /api/auth/login` (argon2 hash — CPU bound).

**Where to run:**
- **CI runner** (GitHub Actions 2 vCPU, 7GB) — free, nhưng network latency làm méo kết quả.
- **VPS chạy k6 cùng lúc với app** — kết quả thật nhất, nhưng CPU conflict.
- **Máy local** (laptop dev 8 vCPU) → VPS qua internet — thực tế nhất.
- **Recommendation:** máy local chạy k6, target VPS URL https://mes.songchau.vn. Đo từ góc end-user.

**Thresholds (pass/fail):**
- P95 < 500ms cho GET endpoint.
- P95 < 1500ms cho POST reservation.
- Error rate < 0.5%.
- Login P95 < 800ms (argon2 cost 19_456KB memory bắt chậm).

**Optimization cycle:**
- Nếu P95 fail → `EXPLAIN ANALYZE` → thêm index → re-run.
- Không phải endpoint nào cũng optimize — 80/20: focus top 3 endpoint chậm nhất.

**Acceptance:**
- 5 endpoint với 100VU chạy 5 phút → report trong `plans/v1.4/load-test-report.md` kèm screenshot Grafana (nếu Monitoring có sẵn).
- Nếu thấy endpoint nào fail → file issue GitHub + plan V1.5 fix.

**Brutal:** Load test trên 2 vCPU VPS với argon2 login ~250ms 1 hash → max theoretically 8 login/s. 100 VU login cùng lúc = certainly fail P95. Đừng hoảng, document "login throughput bound by argon2 cost, scale vertical VPS nếu cần". Argon2 chậm là **feature chống brute-force**, không phải bug.

## §3. Migration 0007 strategy — 3 file chia nhỏ

### 0007a — RLS enable + policies (day 1-3)

```sql
-- 0007a_rls_enable_policies.sql
-- Tạo 4 Postgres role tương ứng 4 app role
CREATE ROLE app_admin NOINHERIT;
CREATE ROLE app_planner NOINHERIT;
CREATE ROLE app_warehouse NOINHERIT;
CREATE ROLE app_operator NOINHERIT;

-- GRANT ở cấp schema
GRANT USAGE ON SCHEMA app TO app_admin, app_planner, app_warehouse, app_operator;
GRANT SELECT ON ALL TABLES IN SCHEMA app TO app_admin, app_planner, app_warehouse, app_operator;
GRANT INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA app TO app_admin;
-- INSERT/UPDATE/DELETE chi tiết per role dưới

-- Enable RLS
ALTER TABLE app.item ENABLE ROW LEVEL SECURITY;
ALTER TABLE app.item FORCE ROW LEVEL SECURITY;
ALTER TABLE app.bom_template ENABLE ROW LEVEL SECURITY;
ALTER TABLE app.bom_template FORCE ROW LEVEL SECURITY;
-- ... 10 bảng

-- Policies 10 bảng × avg 3 policy = 30 policy.
-- Ví dụ item:
CREATE POLICY item_admin_all ON app.item FOR ALL TO app_admin USING (true) WITH CHECK (true);
CREATE POLICY item_read ON app.item FOR SELECT TO app_planner, app_warehouse, app_operator USING (true);
CREATE POLICY item_planner_write ON app.item FOR INSERT, UPDATE TO app_planner USING (true) WITH CHECK (true);
-- Operator không có INSERT/UPDATE → deny by default.
```

## §4. RBAC Matrix cụ thể — 4 role × 12 entity × 6 action

**Ký hiệu:**
- `C` = Create | `R` = Read | `U` = Update | `D` = Delete | `A` = Approve | `T` = Transition (state machine)
- `✓` = cho phép | `—` = deny | `self` = chỉ row do chính user tạo (V1.5+)

### 12 entity:

1. Item (master data)
2. BOM Template
3. BOM Revision
4. Sales Order
5. BOM Snapshot
6. Purchase Request (PR)
7. Purchase Order (PO)
8. Receiving
9. Work Order (WO)
10. Assembly Order (AO)
11. Audit Event
12. User / Role / Session

| **User/Role**      | CRUD   | ✓✓✓✓  | —R——    | —R——     | —R(self) |

**Reading guide:**
- "✓✓✓✓" = Create/Read/Update/Delete đều OK.
- "—✓——" = chỉ Read.
- Cột có 5 ký tự = action thứ 5 thường là Approve hoặc Transition.
- "self" = chỉ row của chính mình (planned V1.5+, V1.4 = all).

### File implementation:

`packages/shared/src/rbac/matrix.ts`:
```typescript
export type RbacAction = 'create' | 'read' | 'update' | 'delete' | 'approve' | 'transition';
export type RbacEntity = 'item' | 'bomTemplate' | 'bomRevision' | 'salesOrder'
  | 'bomSnapshot' | 'pr' | 'po' | 'receiving' | 'wo' | 'ao' | 'audit' | 'user';

type Matrix = Record<Role, Partial<Record<RbacEntity, RbacAction[]>>>;

export const RBAC_MATRIX: Matrix = {
  admin: {
    item: ['create', 'read', 'update', 'delete'],
    // ... full access cho admin
  },
  planner: {
    item: ['create', 'read', 'update'],
    bomTemplate: ['create', 'read', 'update'],
    salesOrder: ['create', 'read', 'update', 'approve'],
    wo: ['create', 'read', 'update', 'transition', 'approve'],
    // ...
  },
  warehouse: {
    item: ['read'],
    receiving: ['create', 'read', 'transition'],
    po: ['read', 'update', 'transition'],
    wo: ['read', 'transition'],
    // ...
  },
  operator: {
    item: ['read'],
    wo: ['read', 'approve', 'transition'],  // approve = confirm scan
    ao: ['read', 'update', 'transition'],
    audit: ['read'],  // self filter V1.5+
  },
};

export function can(roles: Role[], action: RbacAction, entity: RbacEntity): boolean {
  return roles.some(r => RBAC_MATRIX[r]?.[entity]?.includes(action));
}
```

## §5. Risk register — 5 risks

| 1 | **RLS breakage on deploy** — 1 policy quên → production 500 error hàng loạt | 🔴 Critical | Medium | (a) test 40-cell matrix trên staging, (b) feature flag `RLS_ENABLED=true/false` để fast-rollback, (c) migration 0007a chạy trong transaction, fail = rollback auto |

## §7. 15 decisions cần chốt

1. **RLS enforcement strict hay permissive fallback?**
   → **Strict** (FORCE ROW LEVEL SECURITY), feature flag `RLS_ENABLED` cho rollback nhanh nếu prod cháy.

2. **Session revoke immediate (Redis blacklist) hay wait 15 phút (JWT expire)?**
   → **Wait 15 phút**, document rõ trong UI. V1.5+ Redis blacklist nếu need.

3. **Password reset email thật hay chỉ temp password display?**
   → **Admin reset = display temp password**, user forgot = stub log link (V1.4). SMTP thật defer V1.5.

4. **Monitoring self-hosted Prometheus hay Grafana Cloud free?**
   → **Grafana Cloud Free** (không đụng RAM VPS 2GB). Self-hosted khi upgrade VPS 4GB.

5. **Load test máy nào chạy?**
   → **Máy local dev** target VPS qua HTTPS. CI runner defer V1.5 (thêm pipeline).

6. **Rate limit fixed window hay sliding window?**
   → **Fixed 60s window**, đủ V1.4. Sliding phức tạp hơn ~2× LOC.

7. **Rate limit key theo IP hay IP+username?**
   → **IP only** cho login (ngăn brute force cross-account). V1.5+ add IP+username nếu false-positive cao.

8. **Audit rollback tự động hay stub preview SQL?**
   → **Stub preview + copy SQL** (V1.4 an toàn). Auto rollback V2+ với approval flow.

9. **RLS check superuser app connection (BYPASSRLS) hay per-role SET ROLE?**
   → **Superuser app + SET LOCAL role** per transaction. Dễ implement, phù hợp connection pool.

10. **Backup off-site S3 hay Cloudflare R2?**
    → **Cloudflare R2** (10GB free + 10GB egress free). S3 tốn money.

11. **Backup encryption GPG hay age/rage?**
    → **GPG** (đã dùng V1 script). Đừng đổi tool giữa sprint.

12. **Force logout all user khi deploy V1.4?**
    → **Có** — session schema change, invalidate all refresh token một lần. Admin thông báo 24h trước.

13. **User self-reset via email thật hay stub log?**
    → **Stub log + Telegram admin notify** (V1.4). Email SMTP defer V1.5.

14. **12×12 matrix có support "approve" action?**
    → **Có** — PO approve, WO release, ECO approve. Chỉ planner + admin có.

15. **Load test threshold P95 500ms hay 1000ms?**
    → **P95 500ms cho GET, 1500ms cho POST**. Login phí argon2 → 800ms (document exception).

## §8. Defer V2+ (không làm V1.4)

- **SSO / OAuth**: Google Workspace, Azure AD, Keycloak self-hosted → V2 khi > 20 user.
- **MFA**: TOTP (Google Authenticator) + WebAuthn (FIDO2) → V1.6 nếu có audit yêu cầu.
- **Audit forensics tool nâng cao**: timeline viewer, entity graph, anomaly detection → V3.
- **Row-level encryption** (pgcrypto per-column) → V2+ nếu dữ liệu khách hàng nhạy cảm.
- **Data masking** (non-prod dev env che PII) → V2+ khi có dev team > 2 người.
- **SOC2 / ISO27001 compliance artifacts** → khi có customer enterprise yêu cầu.
- **Pentesting thuê ngoài** → V2.5 khi > 50 user.
- **OWASP ZAP / Burp automated scan** → V1.6 optional.
- **Cookie session (bỏ JWT)** — migrate sang Postgres-backed session store → V3 nếu scale.
- **Refresh token rotation với reuse detection** → V1.5+ (V1.4 session schema đã có field `replaced_by_session_id` sẵn, chưa active).
- **Account lockout escalation** (3 lần → 5 phút, 5 lần → 30 phút, 10 lần → admin unlock) → V1.5.
- **IP allowlist per role** (admin chỉ từ VPN) → V2 nếu remote work tăng.
- **Geo-fencing login** (block country khả nghi) → V2+ via Cloudflare WAF rules, không cần code.
- **DDoS protection layer 7** → V2 nếu Cloudflare free plan không đủ.
- **Regulatory export** (GDPR right-to-be-forgotten, data portability) → khi mở rộng EU customer.
